# Going_Scan V2 更新日志




**日期**: 2026-04-04
**核心更新**:  L7 服务指纹识别引擎落地与高并发流水线架构闭环

##  解决的核心痛点 (The Problem)
在将扫描器能力从 L4（传输层无状态探活）向 L7（应用层深度识别）延伸时，面临极大的物理资源冲突与性能陷阱：
- **系统资源隔离 (Resource Contention)**：L4 旁路发包不受操作系统限制，而 L7 依赖原生 Socket 连接。若强行共享并发令牌，极易导致系统文件描述符 (FD) 瞬间耗尽，且慢速 HTTP/SSH 握手会严重阻塞 L4 的极速发包。
- **GC 爆炸与性能黑洞 (GC Spikes)**：传统 L7 扫描器依赖庞大的第三方网络库并在扫描时动态编译正则表达式，导致每次探测产生海量内存碎片。且针对每个端口进行 O(N) 级别的探针遍历会迅速榨干 CPU。
- **TCP 状态污染 (TCP State Pollution)**：在同一 TCP 连接上连续发送多个探针，一旦前者触发 Timeout，底层 Socket 状态即遭破坏，导致后续高价值探针（如 GET 请求）全部失效漏报。
- **异步同步脱节 (Asynchronous Desync)**：极速扫描场景下，主进程生命周期短于 L7 慢速握手周期，导致扫描程序在 Worker 打印出指纹日志前就已强行退出，造成结果严重截断。

##  解决方案与底层实现 (The Solution)
围绕“零 GC、极简、高并发”的架构信仰，重新设计了 L7 生产消费链路，并自主实现了高性能的 Nmap 探针解析器：

1. **绝对解耦的生产者-消费者流水线 (Decoupled Pipeline)**
   - 引入 `RunL7Dispatcher` 作为大总管，将 L4 的无锁环形队列战果引流至带天然反压 (Backpressure) 的 Channel 中。
   - 拉起相互独立的 1000 个 L7 专属 Worker 军团，在不干扰 L4 V12 引擎狂飙的前提下，安全控制 OS 句柄上限。
   - 建立专属的 `l7BufferPool` (4KB)，实现 L7 Socket 读写级别的内存零分配。

2. **零外部依赖的极速 Nmap 引擎 (Zero-Dependency Nmap Parser)**
   - 使用 `//go:embed` 将近 3MB 的官方 `nmap-service-probes` 静态编译入二进制文件，实现单文件部署。
   - **AST 预编译**：在引擎点火期（Init 阶段）一次性完成上万条复杂正则的编译与状态机固化。修复了版本模板正则表达式的局部变量问题，将全局正则编译降至绝对的 0 次运行时分配。
   - **O(1) 探针调度 (Heuristic Routing)**：构建倒排索引 `PortProbeIndex`。针对特性端口（如 80）直接提取精准探针；结合缓存的全局 `NULL` 与 `GetRequest` 探针，彻底消灭 O(N) 遍历。

3. **纯净连接与生命周期管理 (Atomic Probing & Teardown)**
   - 将 TCP `DialTimeout` 逻辑下沉至单个 `tryProbe` 动作内部。每一个探针的发送都伴随一次纯净的全新 TCP 握手与极其严格的绝对超时管理 (`SetDeadline`)，彻底杜绝状态污染。
   - **完美的退出等待链 (The Sync Chain)**：引入原子标识位 `isL4Finished()` 联动闭环机制。严格遵循：L4 物理探测全剧终 -> L7 抽干队列并结束 Worker -> 关闭 ResultStream -> 持久化管家打印落盘并触发 `PersistDone`。做到不漏印任何一条高价值指纹。

4. **边缘格式化与持久化钩子 (Edge Translation & Persistence)**
   - 内部数据流转绝对保持底层 `uint8` 协议格式。仅在数据离开系统、落盘至 JSONL 前的最后一刻，进行 `protocolToStr` 及终端 UI 清洗 (`CleanBanner`)。
   - 预留 SQLite 持久化钩子，为未来的 Tier 2 深度关联漏洞扫描打下坚实地基。

##  使用方式与预期效果 (Usage & Impact)
- **-V 智能降级机制**：新增 `ServiceScan` 选项。未开启时，L7 引擎将进入“光速透传模式”，短路复杂的 Worker 逻辑，直接向持久化层输送 `IP:Port`，确保引擎在纯端口扫描任务下重返极限性能。
- **实战表现**：成功在 L4 并发 1000+ pps 的极速背景下，平稳、精准地识别出 Nginx, OpenSSH, vsftpd 等服务端指纹，兼顾了 Masscan 的狂野速度与 Nmap 的探测精度。

**日期**: 2026-04-03
**核心更新**: 跨平台自适应路由与主动 ARP 嗅探引擎上线

##  解决的核心痛点 (The Problem)
在将引擎从 macOS 本地局域网环境迁移至 Linux 云主机 (如 Vultr VPS) 进行公网无状态高频测试时，触发了 `index out of range [0] with length 0` 的致命 Panic。
- **物理拓扑差异**：云服务商的虚拟化网络隔离机制导致系统 ARP 缓存经常处于冷状态或残缺状态。当底层的 `gopacket/routing` 无法从操作系统内核的路由表中直接读到下一跳 (网关) 的物理 MAC 地址时，会返回 `nil`。
- **逻辑缺陷**：旧版 `resolveAndCacheRoute` 存在“错误吞咽”与零值陷阱，导致空的路由元数据混入发射通道，最终在 Pcap 构建以太网帧进行发包 (`LaunchProbe`) 时发生切片越界崩溃。

##  解决方案与底层实现 (The Solution)
对底层网关寻址模块进行了彻头彻尾的重构，将“被动读取”升级为“主动嗅探”，彻底打通了跨平台兼容性：

1. **防崩溃防御机制 (Defensive Patching)**
    - 在任务分发引擎中引入了严格的错误拦截。当目标 IP 的路由解析彻底失败时，引擎会打印警告并安全跳过该任务，阻断任何脏路由 ID 进入发包队列，确保核心循环的绝对稳定。
2. **跨平台主动 ARP 探测 (Active ARP Resolution)**
    - **机制切换**：当系统路由表命中失败（提供了网关 IP 但缺失硬件 MAC）时，引擎会自动降级并触发 `ActiveARPResolution` 函数。
    - **硬核实现**：调用 `libpcap/npcap` 底层驱动，绕过操作系统协议栈的限制，直接构建并向网关广播纯手工打造的 `ARP Request` 报文。
    - **精准过滤**：通过注入极简的 BPF 过滤器 (`arp and arp[7] = 2 and src host <GatewayIP>`) 并在独立协程中设置 2 秒绝对超时，精确捕获目标网关的 `ARP Reply` 回应，毫秒级提取并固化真实的 DstMAC。

## 使用方式与预期效果 (Usage & Impact)
- **开箱即用，零配置**：开发者和使用者**无需传入任何额外的网卡参数或手动硬编码 MAC 地址**。引擎在启动时会自动探测环境，自适应适配 Windows、macOS 以及各类 Linux 发行版与云原生复杂网络。
- **性能解放**：依托该补丁，Gscanv2 成功跨越了局域网 NAT 设备的物理限制，在 VPS 公网直连环境下顺利点火，实测稳定维持在 **2400+ pps** 的无状态高频吞吐，真正兑现了底层旁路发包的极致性能。

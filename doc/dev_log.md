# Going_Scan V2 更新日志

**日期**: 2026-04-12  
**核心更新**: UAM SQLite 契约层正式落地，GS 全链路主要事实开始进入统一资产状态系统

## 本次完成的核心内容

这一轮的重点已经不再是继续堆叠 GS 的扫描方式，而是把当前 GS 已有成果真正沉淀成可查询、可追溯、可扩展的资产状态层。

### 1. UAM 正式从“设计”变成“实现”

当前已经落地：

- `Run`
- `Host`
- `Endpoint`
- `Observation`
- `Claim`
- `HostProjectionCurrent`
- `EndpointProjectionCurrent`
- `ModuleResult`

并已完成 SQLite schema、索引、视图以及基础写入路径。

这意味着当前仓库里已经不再只有“扫描结果导出”，而是开始具备真正的资产状态骨架。

### 2. GS 已经能把主要事实写入 UAM

当前 GS 进入 UAM 的内容已经覆盖：

- Run 元数据
- 主机探活事实
- L4 端口扫描事实
- L7 服务识别 enrichment
- Host / Endpoint 当前 Projection

特别重要的一点是：

当前不再只是把最终画像塞进 SQLite，而是把主机探活、L4 Observation、L7 Observation、Claim、Projection 都按正式模型推进。

### 3. 查询能力已经形成最小可用闭环

当前已经新增：

- `uam runs`
- `uam hosts`
- `uam endpoints`
- `uam observations`
- `uam report`

其中 `uam report` 已经可以根据 IP 输出一份面向人的综合报告，既展示当前状态，也展示支撑这些状态的扫描事实。

### 4. gping / 专项扫描的 UAM 契约已经准备好

虽然 gping 和专项扫描执行器还没有正式落地，但当前 UAM 侧已经把后续需要的核心入口准备好：

- gping 的 Observation / Claim / Projection 入口
- `manual`
- `override`
- `verification_state`
- module 的 ModuleResult 正式写入入口

这意味着后续真正写 gping 时，已经不需要返工底层数据层。

## 这次刻意没有做的事情

为了保证 UAM 先稳住对象边界和写入路径，这一轮没有去提前做：

- gping 执行器本体
- HTTP / TLS / SSH 等专项扫描执行器
- 更复杂的前端查询界面
- 大规模批量写入优化

这些都属于下一阶段建立在当前契约层上的工作，而不应反过来主导当前结构。

## 当前阶段结论

截至 2026-04-12，Going_Scan 当前已经形成两层正式输出：

1. 扫描层：高速发现 + 服务识别 + 最终画像
2. 状态层：UAM SQLite 资产契约层

当前系统的意义已经从“能扫出来”进一步推进到：

> 能把 GS 当前得到的主要事实沉淀成一个正式、可查询、可追溯的资产状态系统。

**日期**: 2026-04-11  
**核心更新**: 第一阶段末尾收口完成，L4 多协议扫描与可靠 L7 指纹识别正式贯通

## 本次完成的核心内容

本轮不是单点补丁，而是把第一阶段的主链路从 CLI 到结果输出完整收口，确保当前仓库对外呈现的是“可用实现”而不是“半完成草稿”。

### 1. L4 扫描能力完成收口

当前已经接入并验证：

- TCP SYN
- TCP ACK
- TCP Window
- UDP
- ICMP 主机探活
- 多协议组合扫描

任务生成器不再把协议路径硬编码在单一模式里，而是可以根据 profile 生成多协议任务。ACK / Window 也不再只是 CLI 参数，而是进入真实结果流并产生独立语义：

- `tcp-syn` -> `open / closed / filtered`
- `tcp-ack` -> `unfiltered / filtered`
- `tcp-window` -> `open / closed / filtered`
- `udp` -> `open / closed / filtered`

### 2. CLI 入口完成整理

CLI 历经多个版本后，入口参数已做统一收口：

- 默认扫描使用内置高频端口集
- `-F` 与 `--top-ports` 的语义统一
- `--output` 与 `--output-format` 正式接入
- 未真正接入 V2 引擎的参数会被明确拒绝，而不是静默忽略

这样避免了用户传入一堆“看起来支持、实际未生效”的参数后得到误导性结果。

### 3. 事实流与最终画像正式建立

结果输出从“逐条打印”升级为“两层模型”：

1. 终端持续输出事实流
2. 扫描结束后生成最终画像

画像支持：

- `json`
- `yaml`

并保留：

- 原始扫描事实 `facts`
- 聚合状态 `summary_state`
- 当前执行命令元数据
- 当前扫描 profile 列表

这一步的意义是：后续深度扫描不再直接吃零散 stdout，而是吃结构化、可追溯的端口画像。

### 4. L7 服务识别从“可跑”提升到“可靠可接入”

L7 核心链路完成了这几项关键加固：

- `sslports` / `fallback` 调度接入
- `rarity` / `totalwaitms` 解析接入
- 候选探针按优先级排序
- TCP 分段响应累计读取
- 版本模板多分隔符解析
- `$P` / `$SUBST` / `$I` 宏展开
- 结构化指纹输出

当前 L7 输出的不只是：

- `service`
- `banner`

还包括：

- `product`
- `version`
- `info`
- `hostname`
- `os`
- `device`
- `cpes`

这让下一阶段的深度扫描调度终于有了稳定输入。

## 这次刻意没有做的事情

为了保证第一阶段收尾质量，本轮明确拒绝做以下半成品能力：

- `softmatch` 直接参与最终服务定性
- `connect` / `idle` / `protocol` 等未收口扫描方式
- 提前实现 HTTP / FTP / 数据库等深层扫描引擎
- 抢先启用 SQLite 但缺少稳定数据模型

这些能力都会在下一阶段基于当前结构化结果继续推进，而不是在第一阶段末尾埋下可靠性隐患。

## 当前阶段结论

截至 2026-04-11，Going_Scan 已经不再只是 L3/L4 骨架，而是具备以下完整链路：

1. 高性能原始 L4 扫描
2. 多协议事实流结果生成
3. 可信结果驱动的 L7 服务识别
4. 结构化画像输出

第一阶段可以视为完成收尾，下一阶段将从“高速发现”转向“基于结果的深度理解”。

**日期**: 2026-04-04  
**核心更新**: L7 服务指纹识别引擎落地与高并发流水线架构闭环

## 解决的核心痛点

在将扫描器能力从 L4 向 L7 延伸时，核心问题集中在资源隔离、GC 开销、TCP 状态污染以及异步链路收尾上：

- L4 原始发包与 L7 Socket 探测资源模型完全不同
- 动态编译正则与全量探针遍历会造成明显性能黑洞
- 多探针共用连接会污染 TCP 状态
- 主进程生命周期短于 L7 Worker 时，结果容易被截断

## 解决方案与底层实现

### 1. 绝对解耦的生产消费流水线

- 引入 `RunL7Dispatcher`
- 使用专属 L7 worker 池
- 使用 `l7BufferPool` 复用读缓冲

### 2. 内嵌 Nmap 指纹库

- 使用 `//go:embed` 内嵌 `nmap-service-probes`
- 启动期完成探针解析与正则预编译
- 构建按端口查询的探针倒排索引

### 3. 纯净探针连接

- 每个探针基于独立连接执行
- 使用严格超时控制
- 完整等待 L4 -> L7 -> Persist 的关闭链

### 4. 持久化与边缘输出

- 结果统一汇入 `ResultStream`
- 保留 SQLite 挂钩，为下一阶段预留入口

## 使用效果

在开启 `-V` 时，当前系统已经可以在不干扰 L4 扫描主链路的前提下进行服务识别；关闭 `-V` 时，则回到轻量透传模式，保持 L4 的主体吞吐能力。

**日期**: 2026-04-03  
**核心更新**: 跨平台自适应路由与主动 ARP 嗅探引擎上线

## 解决的核心痛点

旧版本在跨平台路由解析时会遇到网关 MAC 获取失败，进而导致底层发包阶段崩溃。

## 解决方案与底层实现

### 1. 防崩溃防御

- 路由解析失败时安全跳过任务
- 阻断脏路由进入发包链路

### 2. 主动 ARP 探测

- 当系统路由表无法直接给出下一跳 MAC 时，触发主动 ARP
- 直接构造 ARP Request
- 利用 BPF 精准过滤 ARP Reply

## 使用效果

引擎不再要求手动硬编码网关 MAC 或做额外平台适配，跨平台环境下的稳定性明显提升。

# Going_Scan 完整功能测试方案

> 适用环境：Linux（root 权限）| 测试目标：C 类或 B 类网段 | 日期：2026-05-04

---

## 一、测试目的

验证 Going_Scan V2 Engine 的全部已实现功能在真实网络环境中的正确性和稳定性，覆盖 GS L4 扫描、L7 服务识别、UAM 资产存储与查询、gping 定向验证的完整闭环。

---

## 二、测试环境要求

### 硬件
- 一台具有 root 权限的 Linux 主机
- 至少一张可用的物理网卡
- 与目标网段处于同一二层网络或可达路由

### 软件
- Go 1.24+ 工具链
- libpcap 开发库（`libpcap-dev` 或 `libpcap-devel`）
- gcc（CGO 编译 SQLite 需要）

### 权限
- 必须以 root 运行所有扫描和 gping raw 路线命令
- 或使用 `setcap cap_net_raw,cap_net_admin=eip ./goscan`

### 目标选择建议
- **C 类网段测试**：选择你拥有测试权限的 /24 网段，如实验室网段 `192.168.x.0/24` 或 `10.x.x.0/24`
- **B 类网段测试**：仅在验证性能和大规模扫描能力时使用，如 `10.0.0.0/16`
- 本文档使用 `TARGET_NET` 作为占位符，实际执行时替换为你的目标网段
- 选取 1-3 个已知开放端口的主机作为 gping 验证目标

## 三、测试结果保存方案

### 目录结构

在项目根目录下创建按时间戳组织的测试结果目录：

```bash
TEST_DIR="test_results/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$TEST_DIR"/{scans,uam,gping,logs}
```

目录说明：

| 子目录 | 用途 |
|--------|------|
| `scans/` | 所有 GS 扫描的画像文件（JSON/YAML） |
| `uam/` | UAM 数据库文件及查询输出 |
| `gping/` | gping 执行输出及模板预览 |
| `logs/` | 各命令的完整终端输出日志 |

### 命名约定

每个测试用例的输出文件按以下格式命名：

```
{阶段}-{用例编号}-{描述}.{扩展名}
```

例如：`01-02-syn-full-port.json` 表示第一阶段第二个用例的 SYN 全端口扫描 JSON 画像。

### 日志记录

所有命令的标准输出和标准错误同时保存到日志文件：

```bash
# 每条命令执行时使用
./goscan scan ... 2>&1 | tee "$TEST_DIR/logs/{用例名}.log"
```

### 第一阶段：L4 扫描模式验证

本阶段验证四种 L4 扫描方法在基础场景下的正确性。所有扫描使用较小的端口范围和较高的计时模板以加快测试速度。

**用例 1-1：TCP SYN 扫描（默认模式）**

```bash
./goscan scan TARGET_NET -p 22,80,443,3306,6379,8080,8443 --syn \
  -T 4 -o "$TEST_DIR/scans/01-01-syn-selected-ports.json" \
  2>&1 | tee "$TEST_DIR/logs/01-01-syn.log"
```

验证点：
- 扫描正常完成，无 panic
- 终端输出实时显示开放的端口
- JSON 画像文件生成且格式有效
- Engine Report 显示 Packets Sent / Matched Replies 计数合理

**用例 1-2：TCP ACK 扫描**

```bash
./goscan scan TARGET_NET -p 22,80,443 --ack \
  -T 4 -o "$TEST_DIR/scans/01-02-ack.json" \
  2>&1 | tee "$TEST_DIR/logs/01-02-ack.log"
```

验证点：
- ACK 扫描正常执行
- 结果中端口状态为 unfiltered 或 filtered（ACK 不区分 open/closed）

**用例 1-3：TCP Window 扫描**

```bash
./goscan scan TARGET_NET -p 22,80,443 --window \
  -T 4 -o "$TEST_DIR/scans/01-03-window.json" \
  2>&1 | tee "$TEST_DIR/logs/01-03-window.log"
```

验证点：
- Window 扫描正常执行
- 部分系统上 Window 扫描可区分 open/closed

**用例 1-4：UDP 扫描**

```bash
./goscan scan TARGET_NET -p 53,123,161,500,1900 --udp \
  -T 4 -o "$TEST_DIR/scans/01-04-udp.json" \
  2>&1 | tee "$TEST_DIR/logs/01-04-udp.log"
```

验证点：
- UDP 扫描正常执行
- 终端输出区分 open/closed/filtered（UDP 扫描通常 filtered 较多）

**用例 1-5：多协议组合扫描**

```bash
./goscan scan TARGET_NET -p 22,53,80,443 --syn --udp \
  -T 4 -o "$TEST_DIR/scans/01-05-multi-protocol.json" \
  2>&1 | tee "$TEST_DIR/logs/01-05-multi-protocol.log"
```

验证点：
- 同时执行 TCP SYN 和 UDP 扫描
- 画像中每个端口包含 tcp-syn 和 udp 两个 profile 的事实

**用例 1-6：多方法组合扫描（SYN + ACK + Window）**

```bash
./goscan scan TARGET_NET -p 80,443 --syn --ack --window \
  -T 4 -o "$TEST_DIR/scans/01-06-triple-tcp.json" \
  2>&1 | tee "$TEST_DIR/logs/01-06-triple-tcp.log"
```

验证点：
- 三种 TCP 方法同时执行
- 每个端口有三种方法的 facts
- summary_state 由多种事实聚合得出

### 第五阶段：性能参数验证

验证 CWND 自适应控制、速率限制、重试机制在实际网络中的表现。

**用例 5-1：计时模板 T0-T5**

```bash
# 分别用 T1（最慢）和 T5（最快）扫描同一个目标，观察差异
for T in 1 3 5; do
  ./goscan scan TARGET_NET -p 1-1000 --syn -T $T \
    -o "$TEST_DIR/scans/05-01-timing-T${T}.json" \
    2>&1 | tee "$TEST_DIR/logs/05-01-timing-T${T}.log"
done
```

验证点：
- T1 最慢但最保守（低并发、长超时）
- T5 最快但可能丢包（高并发、短超时）
- Engine Report 中的并发窗口和 RTO 随 T 值变化

**用例 5-2：发包速率限制**

```bash
./goscan scan TARGET_NET -p 1-1000 --syn -T 4 --max-rate 100 \
  -o "$TEST_DIR/scans/05-02-rate-limit.json" \
  2>&1 | tee "$TEST_DIR/logs/05-02-rate-limit.log"
```

验证点：
- Engine Report 显示 "Effective Send Rate: 100 pps"
- 扫描速度明显受限于速率限制

**用例 5-3：并发度控制**

```bash
./goscan scan TARGET_NET -p 1-1000 --syn -T 4 \
  --min-parallelism 16 --max-parallelism 64 \
  -o "$TEST_DIR/scans/05-03-parallelism.json" \
  2>&1 | tee "$TEST_DIR/logs/05-03-parallelism.log"
```

验证点：
- Engine Report 显示 Parallelism Window: 16/64
- 并发度在 16-64 之间波动

**用例 5-4：重试与超时**

```bash
./goscan scan TARGET_NET -p 1-1000 --syn -T 4 \
  --max-retries 1 --max-rtt-timeout 500 \
  -o "$TEST_DIR/scans/05-04-retry-timeout.json" \
  2>&1 | tee "$TEST_DIR/logs/05-04-retry-timeout.log"
```

验证点：
- 最多重试 1 次
- RTO 上限 500ms

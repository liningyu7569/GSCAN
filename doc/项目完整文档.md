## 1. 项目概述

### 1.1 项目定义

Going_Scan（命令行二进制名 `goscan`）是一个用 Go 语言编写的高性能网络资产扫描与分析平台。它由三个协同工作的子系统组成：

| 子系统 | 全称 | 定位 |
|--------|------|------|
| **GS** | Going_Scan | 高速 L4/L7 资产发现引擎 |
| **gping** | gping | 多路线定向验证探针引擎 |
| **UAM** | Unified Asset Model | 统一资产契约层与知识中枢 |

### 1.2 设计哲学

整个工具链的最终目标不是单纯做"一个更强的扫描器"，而是形成一个完整的资产侦察与确认工作流：

1. GS 高速发现资产
2. UAM 沉淀资产状态
3. gping 对资产进行定向验证、确认、修订
4. 专项扫描对特定协议做深入分析
5. 最终形成可追溯、可确认、可扩展的资产中枢

### 1.3 与传统扫描器的区别

- 传统扫描器：吞吐优先 → 结果粗粒度；或识别优先 → 速度慢
- Going_Scan：L4 高速发现 + L7 精准识别分离，UAM 知识沉淀而非一次性结果输出
- gping 的引入：从"扫完即走"升级为"扫完→确认→沉淀→再验证"的闭环

## 2. 架构总览

### 2.1 三工具关系

`goscan` 是统一的 CLI 入口，提供 `scan`、`gping`、`uam` 三个子命令，分别对应三个子系统。三个子系统共享同一个 SQLite 数据库（uam.db）作为数据交换中枢。

**GS（scan 命令）**负责 L4 无状态扫描和 L7 服务识别，产出主机、端口、服务等初步资产事实。扫描过程中通过 UAM 钩子（uam_hook.go）异步写入 Observation、Claim 和 Projection。

**gping** 负责定向验证，可以从 UAM 读取资产上下文作为目标来源，执行 Raw/Stack/App 三路线探测。执行结果通过 GPingIngester 同步写入 UAM 的 Observation 和 Claim，触发 Projection 刷新。

**UAM** 作为数据中枢，以 SQLite 存储所有资产状态。它不发包、不探测，只负责资产身份管理、观察存储、Claim 归一、投影维护和查询报告。

三者构成的协作闭环：GS 广撒网发现资产 → UAM 沉淀和管理资产状态 → gping 精确认和修订资产 → 结果回写 UAM 形成更新后的资产视图。

## 3. GS 引擎详解

### 3.1 核心数据结构

#### EmissionTask（8 字节压缩扫描任务）

```go
type EmissionTask struct {
    TargetIP        uint32 // 目标 IPv4（大端序）
    TargetPort      uint16 // 目标端口
    RouteID         uint16 // 全局路由表索引
    Protocol        uint8  // IPPROTO_TCP / UDP / ICMP
    ScanFlags       uint8  // TCP 标志位（SYN=0x02, ACK=0x10...）
    ScanKind        uint8  // 扫描方法：SYN/ACK/Window/UDP
    IsHostDiscovery bool   // 是否为探活任务
}
```

恰好 8 字节（64 bits），在并发通道中高效传递。

#### PacketTensor（32 位回包特征）

```go
type PacketTensor uint32

// 位布局
// bits 0-7   : TCP Flags / ICMP Type / UDP 虚拟标志
// bits 8-15  : IP TTL
// bits 16-19 : TCP Window 量化值 / ICMP Code
// bits 20-27 : IP Protocol
// 0xFFFFFFFF : 超时魔术字
```

`ExtractTensor()` 在收包器中 O(1) 完成报文坍缩。配套方法 `IsTCPStateOpen()`、`IsTCPStateClosed()`、`IsTCPFiltered()`、`IsUDPStateOpen()`、`IsUDPStateClosed()`、`IsHostAlive()` 等全部用纯位运算实现状态判定。

#### RouteMeta（L2/L3 物理链路信息）

```go
type RouteMeta struct {
    SrcIP  uint32   // 发包源 IP
    SrcMAC [6]byte  // 本机网卡 MAC
    DstMAC [6]byte  // 下一跳 MAC
}
```

通过 `GlobalRouteCache` 切片 + `RouteID` 索引实现 O(1) 访问，支持多网卡/多网关场景。

### 3.2 Engine 运行时

Engine 启动时创建三个后台 goroutine：

- **runSendPacer**：速率控制令牌生成器，按配置的 pps 速率向 sendTokens 通道补充令牌
- **RunDispatcher**：pcap 收包循环，持续从网卡读取回包，完成 Channel ID 解码、目标校验、张量坍缩、结果投递
- **RunL7Dispatcher**：L7 服务识别消费循环，从 GlobalResultBuffer 获取开放端口结果，分发给 worker 池

主循环不断从 TaskGenerator 获取任务批次，对每个任务：

1. 从 Tokens 通道获取令牌（CWND 并发控制）
2. 从 FreeIDs 通道获取可用的 channelID
3. 启动 goroutine 执行 LaunchProbe

LaunchProbe 的执行流程：

- 从 GlobalRouteCache 获取路由信息（源 IP、MAC 地址）
- 通过 waitSendTurn 获取速率控制令牌
- 调用 BuildIntoBuffer 在池化内存上构造完整报文
- 通过 pcap.WritePacketData 发送数据包
- 进入 select 等待：channels[id] 收到 PacketTensor 则成功，timer 超时则重试或标记 filtered
- 成功时调用 increaseCWND + updateRTO，失败时调用 decreaseCWND
- 通过 evaluateTaskResult 判定端口状态，emitScanResult 写入结果缓冲

### 3.3 CWND 自适应并发

- **increaseCWND()**：每次成功收到回包，尝试向 Tokens 通道放入一个令牌，`currentCapacity +1`
- **decreaseCWND()**：超时时，尝试从 Tokens 通道取出 `currentCapacity/2` 个令牌，但不低于 MinParallelism
- **updateRTO(rtt)**：指数平滑 `newSRTT = (srtt*7 + rtt)/8`，`newRTO = newSRTT*3`，范围 [50ms, MaxRTTTimeout]

### 3.4 发包流程

`BuildIntoBuffer()` → 直接从 `sync.Pool` 获取 128 字节缓冲区，构造完整以太网/IP/TCP 或 UDP 或 ICMP 报文。

**Channel ID 编码**：将通道 ID 编码到 TCP/UDP 源端口号中：
```
源端口 = BASE_PORT + channelID
```
收包时从回包的目的端口直接解码出 `channelID`，实现 O(1) 探针匹配。

### 3.5 收包与匹配（RunDispatcher）

RunDispatcher 在后台循环执行以下流程：

1. 调用 `pcapHandle.ZeroCopyReadPacketData()` 从网卡零拷贝读取原始数据包
2. 检查包有效性：以太网类型必须为 0x0800 (IPv4)、包长度 ≥ 34、IP 头长度必须完整
3. 提取 IP 协议号（packetData[23]），根据协议类型分派：
   - TCP/UDP：从传输层头部解码 channelID（目的端口号反查），提取 srcIP 和 srcPort
   - ICMP：调用 decodeICMPMatch() 解析 ICMP 错误消息中内嵌的原始 IP 包，从中提取 channelID、srcIP、srcPort
   - 其他协议：跳过
4. 构造校验码 `(srcIP << 16) | srcPort`，与 `Targets[channelID]` 中原子存储的期望值比对，不匹配则丢弃
5. 调用 ExtractTensor() 将 IPv4 头部和传输层头部坍缩为 32 位 PacketTensor
6. 非阻塞写入 `e.Channels[channelID]`，若通道满则计入 MetricDispatchDrops

### 3.6 L7 服务识别（RunL7Dispatcher）

L7 引擎有两种模式：

**Passthrough 模式（无 -V）**：直接将 L4 结果透传，标记为 `service=""`。

**完整服务检测模式（-V）**：
- 从 `GlobalResultBuffer` 消费 L4 开放结果
- 1000 个 worker goroutine 从 `l7TaskQueue` 取任务
- 每个任务根据端口索引选择候选 nmap 探针
- 建立 TCP 连接 → 发送探针 → 累积读取响应
- 用预编译正则匹配提取 service/product/version/info/hostname/os/device/cpes
- 结果写回 `GlobalResultBuffer`

**nmap-service-probes 集成**：
- 启动时解析 `nmap-service-probes` 文件
- 提取所有 Probe（探针字符串）、Match（正则 + 服务信息）、ports/sslports/fallback 指令
- 构建 port→probes 索引，按 rarity 排序
- 所有正则预编译，零运行时开销

### 3.7 UAM 钩子（uam_hook.go）

扫描过程中异步写入 UAM SQLite：

- **事件队列**：65536 容量 channel
- **GSIngester**：后台 goroutine 消费事件，写入 SQLite
  - `IngestResult()`：L4 扫描结果 → Observation + Claims + Projection 刷新
  - `IngestServiceResult()`：L7 识别结果 → 附加 Claim（不覆盖 L4 端口状态）
  - `IngestDiscovery()`：主机探活 → Host 可达性标记
- **PersistDone**：写入完成信号 channel，确保主流程等待所有 UAM 写入完成

### 3.8 结果画像（report.go）

扫描结束后，将所有事实聚合为 `PortraitReport`：

```json
{
  "command": "goscan scan 192.168.1.10 --syn -V",
```

## 4. gping 探针引擎详解

### 4.1 设计定位

gping 不是 GS 的连续体或后处理器，而是一个独立的多路线探针引擎：

1. **独立工具身份**：可以不依赖 UAM，直接对 IP/URL 执行探测
2. **UAM 确认器身份**：读取 UAM 资产上下文，执行验证后回写结论

### 4.2 统一生命周期

所有动作共享同一生命周期：

```
Resolve → Plan → Prepare → Execute → Interpret → Emit
```

- **Resolve**：解析目标来源（URL / 字面 IP / UAM 端点 / UAM 查询）
- **Plan**：构建 ActionUnit 列表（模板展开或 CLI 参数）
- **Prepare**：构建具体执行参数（`preparedRun`）
- **Execute**：路由到 Raw / Stack / App 执行器
- **Interpret**：将执行证据转为标准化 Claims
- **Emit**：终端输出 / JSON / UAM 写入

### 4.3 三路线执行器

#### Raw 路线（raw.go, ~849 行）

绕过 OS 协议栈，直接构造和发送原始数据包。

**TCP 探测（executeRawTCP）**：
- 通过 gopacket 构造 Ethernet / IP / TCP 层
- 支持自定义 TCP Flags、TTL、Window Size、数据长度
- 打开 pcap handle + BPF 过滤器只收相关回包
- `rawTCPFlags` 结构体提供完整 TCP 标志位解析
- 支持 strict 模式（严格执行）和 probe 模式（探测实验）

**ICMP 探测（executeRawICMP）**：
- 构造 Ethernet / IP / ICMP Echo Request
- 接收 Echo Reply 或 ICMP 错误
- 判断主机可达性

**特殊功能**：
- `corruptTransportChecksum()`：发送错误校验和包（badsum 测试）
- `openRawHandle()`：带 BPF 过滤的 pcap 打开
- 完整的超时与重试机制

#### Stack 路线（stack.go, ~289 行）

使用 OS 协议栈进行真实通信。

**TCP Connect（executeTCPConnect）**：
- 标准 `net.DialTimeout` TCP 连接
- 通过错误类型分类：ECONNREFUSED → closed、timeout → filtered、成功 → open

**TLS 握手（executeTLSHandshake）**：
- TCP 连接 + TLS ClientHello
- 提取服务器证书信息：Subject、Issuer、SANs、ALPN、TLS 版本
- 自定义 TLS Config 支持跳过验证

**Banner 读取（executeBannerRead）**：
- TCP 连接 + `SetReadDeadline`
- 读取初始 banner 字节，解析 product/version

#### App 路线（app.go + 7 个适配器）

应用层协议适配器，通过 `AppAdapter` 接口统一：

```go
type AppAdapter interface {
    Name() string
    Capabilities() AdapterCapabilities
    Execute(ctx context.Context, req AppRequest) (AppResult, error)
}
```

| 适配器 | 文件 | 支持方法 | 行数 |
|--------|------|----------|------|
| HTTP | `app_adapter_http.go` | http-head, http-get, http-post | ~166 |
| DNS | `app_adapter_dns.go` | dns-query（A/NS/CNAME/TXT/AAAA） | ~437 |
| SSH | `app_adapter_ssh.go` | ssh-banner, ssh-kexinit, ssh-hostkey | ~582 |
| MySQL | `app_adapter_mysql.go` | mysql-greeting, mysql-capabilities, mysql-starttls | ~462 |
| Redis | `app_adapter_redis.go` | redis-ping, redis-info-server, redis-info-replication | ~336 |
| FTP | `app_adapter_ftp.go` | ftp-banner, ftp-feat, ftp-auth-tls | ~403 |
| SMTP | `app_adapter_smtp.go` | smtp-banner, smtp-ehlo, smtp-starttls | ~366 |

**适配器特色功能**：
- SSH：完整二进制 SSH 协议实现 + ECDH 密钥交换 + SHA256 主机密钥指纹
- MySQL：协议版本解析 + Server Version + Capability Flags + MariaDB/Percona flavor 检测
- Redis：RESP 协议实现 + INFO 命令解析 + 认证检测
- DNS：UDP/TCP 双传输 + 完整 DNS 线格式编解码
- SMTP：STARTTLS 升级 + TLS 证书提取
- FTP：RFC 959 多行回复解析 + AUTH TLS

### 4.5 解释器（interpret.go, ~419 行）

`buildClaimsFromEvidence()` 将执行证据转换为 UAM Claims：

- 24+ 方法各自对应的 Claim 生成逻辑
- TCP 方法 → 端口状态 Claim（open/closed/filtered）
- ICMP 方法 → 主机可达性 Claim
- TLS 握手 → TLS 版本/证书 Subject/Issuer Claim
- HTTP → 服务器产品/HTML Title Claim
- DNS → DNS 服务器/解析结果 Claim
- SSH → Banner/主机密钥指纹 Claim
- MySQL → 版本/Flavor Claim
- Redis → 版本/角色（master/slave）Claim
- FTP/SMTP → Banner/产品/版本 Claim

### 4.6 目标解析器（resolver.go, ~317 行）

四种目标来源：

1. **URL 解析**：`https://1.2.3.4/path` → TargetContext{IP, Port, Scheme, Path, SNI}
2. **字面目标**：`--ip 1.2.3.4 --port 443` → 直接构造
3. **UAM 端点 ID**：`--uam-endpoint <id>` → 从 SQLite 查询
4. **UAM 筛选查询**：`--uam-service https --pick-index 1` → 查询端点列表 + 按索引选取

支持 `tryHydrateFromUAM()`：从 UAM 补充上文（service/product/verification_state）。

### 4.7 辅助功能

- **candidates**：列出 UAM 中符合筛选条件的端点，每个候选自动推荐模板
- **preview**：完成 Resolve → Plan，输出将执行的动作列表，但不实际执行
- **history**：查询 gping 历史执行记录，展示 observation 和 endpoint 状态变化
- **suggestions**：基于 port/protocol/service/product 启发式推荐模板

## 5. UAM 统一资产模型详解

### 5.1 五层对象模型

UAM 采用严格的五层分层设计，自底向上依次为：

**第一层：Identity（身份层）**
- Host：主机身份，当前阶段以 IP 为唯一标识（host_id = 'host:' + ip）
- Endpoint：端点身份，以 IP + Protocol + Port 为复合标识
- 身份对象只增不改，记录首次和最后发现时间

**第二层：Observation（观察层）**
- 记录具体工具在具体时间看到了什么原始事实
- 来源包括：GS 扫描结果、gping 动作证据、Module 专项结果
- Observation 只追加、不可变，保留完整的 data_json 原始数据

**第三层：Claim（断言层）**
- 将 Observation 归一化为统一的资产断言
- 端口状态：open / closed / filtered / unfiltered
- 服务识别：service / product / version / info / hostname / os / device / cpes
- 主机可达性：reachable / unreachable
- 人工确认状态：none / pending / confirmed / overridden
- Claim 可被更高优先级的后续 Claim 覆盖

**第四层：Projection（投影层）**
- 由 Claim 推导的当前最佳视图，类似物化视图
- HostProjectionCurrent：主机级别的当前状态（可达性等）
- EndpointProjectionCurrent：端点级别的当前状态（端口状态、服务信息、验证状态等）
- 每次新 Claim 写入后增量刷新，不重新计算全量

**第五层：Extension（扩展层）**
- ModuleResult：专项扫描的深结果挂接点
- 模块专属字段进入此层，不污染核心资产模型
- 通过 module_name / result_type / result_json 灵活承载任意结构

### 5.2 Claim 优先级体系

```go
const (
    PriorityInferred  = 1  // 程序推断（如开放端口推断服务类型）
    PriorityObserved  = 2  // 工具观察（GS/gping 自动探测结果）
    PriorityManual    = 3  // 人工确认（gping --assert confirmed）
    PriorityOverride  = 4  // 人工覆盖（gping --assert overridden）
)
```

投影刷新规则：
- 多个 Claim 竞争同一字段时，优先级高的覆盖优先级低的
- 同优先级时，时间戳新的覆盖旧的
- `override` 和 `manual` 优先级的 Claim 不会被后续自动扫描的结果覆盖

## 7. 数据流与生命周期

### 7.1 GS 扫描完整数据流

**第一阶段：任务生成**

IP 迭代器（支持单 IP、CIDR、范围）结合端口列表和扫描 Profile（tcp-syn/tcp-ack/tcp-window/udp），由 TaskGenerator 批量生成 EmissionTask。每个 EmissionTask 精确 8 字节，包含目标 IP、端口、路由索引、协议、标志位、扫描类型等信息。

**第二阶段：L4 探测**

Engine.Run() 主循环消费 TaskGenerator 产出的任务批次。每个任务经过 CWND 并发控制（获取 Token）、Channel ID 分配（获取 FreeID）、速率控制（获取 sendToken），然后启动 goroutine 执行 LaunchProbe：

- BuildIntoBuffer() 在池化内存上直接构造以太网/IP/TCP（或 UDP/ICMP）报文
- 通过 pcap.WritePacketData() 从网卡发出
- 进入 select 等待回包（channels[id]）或超时（timer）
- 成功回包经过 ExtractTensor() 坍缩为 32 位 PacketTensor
- evaluateTaskResult() 将张量映射为端口状态（open/closed/filtered）
- 结果通过 emitScanResult() 写入 GlobalResultBuffer
- 若指定了 --uam-db，同时通过 uam_hook 异步推送到 GSIngester

RunDispatcher() 在后台持续收包：pcap 零拷贝读取 → 包有效性检查 → Channel ID 解码 → 目标校验 → 张量坍缩 → 写入对应 goroutine 的 channels[id]。

**第三阶段：L7 服务识别**

RunL7Dispatcher() 从 GlobalResultBuffer 消费 L4 开放端口结果。若未开启 -V，直接透传标记为 service=""。若开启 -V：

- 1000 个 worker goroutine 从 l7TaskQueue 获取任务
- 根据端口查索引选择候选 nmap 探针
- 建立 TCP 连接，发送探针字符串，累积读取响应
- 用预编译正则匹配提取 service/product/version/info/hostname/os/device/cpes
- 结果写回 GlobalResultBuffer，若指定了 --uam-db 则推送到 GSIngester（仅写服务信息，不覆盖 L4 端口状态）

**第四阶段：画像输出**

扫描结束后，report.go 将所有事实按 Host → Endpoint 聚合，计算 summary_state（多 profile 一致取共识，不一致标记 indeterminate），生成 PortraitReport 导出为 JSON 或 YAML 文件。

### 7.2 UAM 写入路径

不同子系统的写入路径共享同一个底层模式：

**写入入口**：

- scan 命令通过 uam_hook.go 的事件队列（65536 容量 channel）异步写入
- gping 命令通过 ingest_gping.go 同步写入
- module 命令通过 ingest_module.go 同步写入

**写入流程（contractIngester 基类）**：

1. BeginTx() 开启 SQLite 事务
2. 确保 Host 和 Endpoint 身份记录存在（EnsureHost / EnsureEndpoint，INSERT OR IGNORE 语义）
3. InsertObservation() 写入原始观察记录
4. InsertClaims() 写入归一断言（可多条）
5. 若有模块专属数据，InsertModuleResult() 写入扩展结果
6. Commit() 提交事务
7. refreshProjections() 刷新受影响字段的投影：
   - LoadProjection() 加载当前投影（若无则创建空投影）
   - 查询该 Subject 的所有相关 Claim（按优先级和时间排序）
   - 依次检查 ShouldApply()，应用符合条件的 Claim
   - SaveProjection() 以 UPSERT 语义保存更新后的投影

### 7.3 gping 执行生命周期

gping 的所有执行路径共享统一的六阶段生命周期：

**第一阶段：Resolve（目标解析）**

ResolveTarget() 根据输入类型分派到四个解析器：
- URL 输入 → resolveFromURL()：解析 scheme/host/port/path，将主机名解析为 IPv4
- 字面 IP → 直接构造 TargetContext
- UAM 端点 ID → resolveFromUAMEndpoint()：从 SQLite 查询端点详情
- UAM 筛选条件 → resolveFromUAMFilters()：按 service/port/protocol/verification 查询端点列表，通过 pick-index 或 pick-first 选择目标
- 若指定了 --uam-db，tryHydrateFromUAM() 补充 service/product/verification_state 等上下文

**第二阶段：Plan（动作规划）**

Plan() 根据输入类型构建 ActionUnit 列表：
- 模板执行 → planFromTemplate()：加载模板 YAML，展开 ${var} 变量（base + defaults + CLI 覆写），将 workflow 中的每个 step 编译为 ActionUnit，评估 when 条件
- CLI 参数 → buildSingleAction()：根据 --method/--route/--port 等标志构建单个 ActionUnit

**第三阶段：Prepare（执行准备）**

构建 preparedRun 上下文，包含目标信息、动作列表、变量解析结果、预览值。

**第四阶段：Execute（执行探测）**

对每个 ActionUnit，executeAction() 按 route 字段分派：
- route=raw → executeRaw() → executeRawTCP() 或 executeRawICMP()
- route=stack → executeStack() → executeTCPConnect() / executeTLSHandshake() / executeBannerRead()
- route=app → executeApp() → 适配器分发到 HTTP/DNS/SSH/MySQL/Redis/FTP/SMTP

**第五阶段：Interpret（证据解释）**

interpretEvidence() 将 ExecutionReport 中的 routeEvidence 转换为标准化 UAM Claims：
- actionTypeForMethod() 将方法名映射为 UAM ActionType（reach/collect/handshake/request/probe）
- buildClaimsFromEvidence() 根据方法和路线生成 port_state / service / reachability / tls / banner 等 Claim

**第六阶段：Emit（结果输出）**

- 终端：打印执行摘要和 Claim 列表
- JSON：输出完整的 RunResult
- UAM：若指定了 --uam-db，writeReportsToUAM() 通过 GPingIngester 将 Observation + Claims 写入 SQLite，触发 Projection 刷新
- 同时处理 --assert 标志对应的 operator assertions（verification_state / override_service_name）

## 9. 关键设计决策

### 9.1 为什么不使用 connect() 作为主扫描路径

OS 的 `connect()` 会触发完整的 TCP 状态机，内核会自行重传、维护连接状态、限制并发连接数。对于大规模端口扫描，这些行为会严重限制吞吐和可控性。

Going_Scan 选择直接构造原始报文 + pcap 收包，完全掌控发包节奏、重传策略和并发窗口。

### 9.2 为什么使用 PacketTensor 而不是完整的协议解析

完整的逐层协议解析（以太网 → IP → TCP → 应用层）在高速收包场景下是性能瓶颈。PacketTensor 用 32 位编码提取关键特征（协议、标志位、TTL、窗口大小），所有状态判定通过 O(1) 位运算完成，无需分配、无需分支预测失败。

### 9.3 为什么 UAM 要分离 Observation 和 Claim

Observation 记录"在什么时间、用什么工具、看到了什么"——这是原始事实。
Claim 记录"这意味着什么资产断言"——这是归一结论。

分离的目的：
- 同一 Observation 可以产生多个 Claim
- 不同工具的 Observation 可以产生语义统一的 Claim
- Claim 可以被后续更高优先级的观察覆盖
- Observation 永远不可变（只追加），Claim 可以被覆盖

### 9.4 为什么 UAM 要有 Projection 层

如果没有 Projection，每次查询"当前端口状态是什么"都需要扫描所有 Claim 重新计算。Projection 是物化视图，始终反映当前最佳状态。每当有新 Claim 写入，只刷新受影响字段的 Projection，而不是重新计算全量。

### 9.5 为什么 gping 要有三条路线

- **Raw**：绕过协议栈，精确控制每一个包字段（TTL、Flags、Window、校验和），适合防火墙测试、畸形包实验
- **Stack**：使用 OS 协议栈，真实通信行为，适合确认服务是否真的可达
- **App**：应用层语义，适合确认"这个端口上跑的确实是 HTTPS，证书是什么"

三条路线解决不同层次的问题，不可互相替代。

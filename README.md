# Going_Scan (Gscanv2) - L3/L4 Core Engine

**高性能、零分配、异步无锁的分布式扫描引擎核心**

> **当前状态：正在施工 (Work In Progress)**
> 本仓库目前展示的是 Going_Scan 架构中的 L3/L4 探测引擎底座。它追求极致的并发性能与极低的 GC 开销。目前引擎的基础收发流水线已构建完毕，但在部分极端网络边界情况和自适应限流方向仍存在缺陷，处于积极施工状态。

## 核心发包机制 (How It Works)

Gscanv2 彻底抛弃了传统的操作系统 Socket API（如 `connect()`），采用了**内核旁路 (Kernel Bypass)** 与 **收发解耦** 的极致异步架构：

1. **L2/L3 原始报文注入**：底层强依赖 `pcap`，通过构建以太网帧、IPv4首部和传输层（TCP/UDP/ICMP）伪首部，直接向网卡发送 Raw Packet。
2. **零内存分配 (Zero-Allocation)**：
    * 发包端利用 `sync.Pool` 复用 `*[]byte` 底层数组，通过指针偏移直接装填协议头部，彻底阻断逃逸分析。
    * 收包端利用 `ZeroCopyReadPacketData()`，避免内核态到用户态的内存拷贝。
3. **O(1) 状态坍缩 (PacketTensor)**：
    * 将庞大的协议解析逻辑压缩为一个 32 位的 `PacketTensor` (张量)。只用几条位运算和掩码，即可在纳秒级判定端口开放状态或主机存活状态。
4. **收发绝对解耦**：
    * `LaunchProbe` 协程池只负责组装和发射弹药。
    * `RunDispatcher` 死循环作为唯一的物理接收端，通过精确的 `channelID` 和端口/IP验证码，将回包张量 O(1) 路由回对应的探针，互不阻塞。

## 架构特色 (Highlights)

* **平滑探活流水线 (Streaming Host Discovery)**：
  打破 Nmap 传统的“两阶段隔离”（先探活后扫端口）弊端。引入**蓄水池机制 (HD Reservoir)**，一旦收到探活回包，瞬间将其推入高优先级队列，实现“边探活、边扫描”，彻底消除 I/O 等待迟滞。
* **极致无锁化 (Lock-Free Design)**：
    * 全局数据流转采用泛型化的 `LockFreeRingBuffer`，利用底层的 CAS 操作和 Cache Line 伪共享隔离（Padding），承载百万级 PPS 流转。
    * 探活去重采用 **分片 Map (Sharded Map)**，将全局锁竞争概率降至极低。
* **分片指标池 (Sharded Metrics)**：
  抛弃全局 `atomic.Add`，采用分片对齐机制统计发包量、命中率和丢包率，消除极限并发下的 CPU 缓存行争用。

## 现存局限与正在施工的方向

当前引擎在追求物理硬件处理极限的同时，在以下几个方向仍存在未完善的缺陷，正在积极排查与施工中：

1. **网络边界感知与自适应拥塞控制**：在面对脆弱的路由器 NAT 表或低带宽链路时，缺乏像 BBR 或 TCP 拥塞算法那样的智能降速与背压反馈机制，容易造成链路阻塞或静默丢包。
2. **底层协议兼容性**：L2 协议解析目前存在一定的硬编码偏移量，在处理 VLAN 等复杂链路层环境时可能存在解析偏差。
3. **高阶探针缺失**：针对 UDP 探测等需要应用层 Payload 的场景，目前仍缺乏完备的探针库支持。
